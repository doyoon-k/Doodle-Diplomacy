LLM Pipeline Practical Tutorial

Audience: Unity developers who want to attach a local LLM pipeline to gameplay, generate structured JSON from game state, and apply it safely at runtime.

1) Tutorial goals

After this tutorial, you should be able to:

  1. Understand how each LlmGenerationProfile field affects behavior.
  2. Build and validate step graphs in Prompt Pipeline Editor.
  3. Set up a scene so the pipeline runs correctly in Play mode.
  4. Map LLM output state into gameplay systems (stats, skills, descriptions).

2) LlmGenerationProfile explained

Create profiles via Create > LLM > Generation Profile.

2-1. Basic settings

FieldMeaningRecommendation
model Path to the GGUF model file used by this profile. Use a path under Assets/StreamingAssets/Models.
systemPromptTemplate System-level instruction template. Supports state placeholders like {{key}}. Keep role, rules, and output contract stable here.
stream, keepAlive Service interoperability options. Keep defaults unless your service flow requires changes.

2-2. Model params

FieldMeaningPractical note
temperatureCreativity/randomnessFor strict JSON stability, start around 0.2 to 0.7
top_pNucleus sampling cutoffUsually 0.8 to 0.95
top_kToken candidate limitLower can be more conservative
num_predictMax generated tokens0 uses runtime default (-1 behavior)
repeat_penaltyRepetition suppressionToo high can make output unnatural

gpuLayerCountNumber of GPU-offloaded layersTune based on backend + VRAM
threadsCPU thread count0 = auto

2-4. JSON output fields

Editing jsonFields updates the output schema contract used by JSON pipeline steps. Better schema constraints generally produce more stable and parseable responses.

  • Keep nested depth reasonable for maintainability.
  • Use clear names and explicit numeric/text constraints when possible.
  • Separate profiles by purpose (for example, stats update vs skill generation).
Best practice: split profiles by task.
Example: use one profile for stat evolution JSON and another for skill design JSON.

3) Pipeline Editor workflow

Menu: Window > LLM > Prompt Pipeline Editor

3-1. Basic flow

  1. Create a new pipeline asset with New Asset.
  2. Add nodes with Add Step.
  3. Connect Exec Out -> Exec In to define execution order.
  4. For each step, set Step Kind, profile, and prompt template.
  5. Use the Simulation panel on the right to run with test state values.
  6. Review Step Log, then save.

3-2. Choosing step kinds

  • JsonLlm: preferred default for structured outputs.
  • CompletionLlm: plain text generation only.
  • CustomLink: deterministic C# logic between LLM steps.

For detailed UI field behavior, see PipelineEditorGuide.md.

4) Scene setup checklist

This is the minimum setup required to make the pipeline run correctly in Play mode.

4-1. Required assets

  1. One GGUF model file (for example Assets/StreamingAssets/Models/qwen3-4b-q4_k_m.gguf)
  2. One or more LlmGenerationProfile assets
  3. One PromptPipelineAsset asset

4-2. Required components

GameObjectComponentNotes
LLM Host (recommended single object) RuntimeLlamaSharpService Keep registerOnAwake and persistAcrossScenes enabled.
Same object or separate runner object GamePipelineRunner Assign runtime service reference if not on same object.
Gameplay integration object Example: ItemManager Assign the target pipeline asset.

4-3. Required Setup Wizard actions

  1. Open Tools > LLM Pipeline > Setup Wizard.
  2. Click Use Recommended.
  3. Click Install/Update Dependencies.
  4. Click Apply Backend Configuration.
  5. Select model and click Apply Model To All Profiles.
  6. Click Validate Setup.

5) Runtime execution flow

  1. Gameplay script builds an input state dictionary.
  2. GamePipelineRunner.RunPipeline(...) is called.
  3. Pipeline steps execute in order.
  4. JSON/text output is merged into final state.
  5. Final callback applies state to game systems.
var state = new Dictionary<string, string>
{
    ["item_name"] = item.itemName,
    ["item_desc"] = item.description,
    ["character_desc"] = playerStats.characterDescription
};

GamePipelineRunner.Instance.RunPipeline(pipelineAsset, state, finalState =>
{
    ApplyResultToGameplay(finalState);
});

6) Gameplay integration patterns

Pattern A: stat evolution

  • Input: current stats + absorbed item theme
  • Output: additive/multiplicative stat modifiers in JSON
  • Apply: update stat model and UI

Pattern B: ability synthesis

  • Input: primitive move list + character fantasy constraints
  • Output: new skill composition JSON
  • Apply: convert to runtime skill data and register in skill manager

Pattern C: narrative flavor update

  • Input: previous character description + latest event
  • Output: updated short identity/fantasy text
  • Apply: HUD/lore panel refresh

7) What the sample demo does

  1. Pick up an item in scene.
  2. ItemManager composes context state.
  3. Pipeline generates evolved stats and a new skill in JSON.
  4. Result applies to gameplay (stats and equipped skill set).
  5. UI immediately displays evolved values.

Default controls (sample scene):

  • Move: A/D
  • Jump: Space
  • Attack: J
  • Shoot: K
  • Skills: Q/E
  • Use item: 4
  • Swap item: T

8) Packaging and deployment tips

  1. Ship model files separately from code package when possible.
  2. Let users choose one backend per machine (avoid mixed backend conflicts).
  3. Run setup validation before playtest/build.
  4. For exported demos, document input mode and render pipeline assumptions.
If users report native load failures, first verify backend selection, NuGet restore state, and duplicate plugin import settings.
